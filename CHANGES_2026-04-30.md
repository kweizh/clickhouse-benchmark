# ClickHouse Benchmark — Changes (2026-04-30)

Triggered by an investigation into why the same ~6 tasks fail on every model in
the `clickhouse-tasks` dataset. Across the 5 most recent runs (one per model:
`zai/glm-4.7`, `google/gemini-3-flash`, `google/gemini-3.1-pro`,
`anthropic/claude-4-6-sonnet`, `openai/gpt-5.2-codex`), 4 tasks scored 0.0 on
every model and 2 hit `RuntimeError` during environment setup on every model.

## 1. Rename: `clickhouse-tasks/` → `clickhouse-benchmark/`

The dataset is now the canonical benchmark suite. Job results from
`jobs/2026-04-30__*` were also moved into `clickhouse-benchmark/jobs/` so each
model's run lives next to the tasks it ran against.

`.gitignore` previously matched `jobs/` at any depth (so benchmark job logs
were never tracked). Added a carve-out:

```
jobs/
!clickhouse-benchmark/jobs/
!clickhouse-benchmark/jobs/**
```

Top-level `jobs/` (the agent-side scratch dir) stays ignored.

## 2. Tasks dropped — environment broken (2)

These hit `RuntimeError` during `_setup_environment` on every trial:

| Task | Cause |
|---|---|
| `clickhouse_metrics_dashboard` | `RUN agent-browser install` in Dockerfile fails on Linux ARM64: `Chrome for Testing does not provide Linux ARM64 builds`. |
| `system_metrics_dashboard` | Same — depends on `agent-browser`. |

The image build itself never completes, so the agent never sees the task. Both
removed entirely.

## 3. Tasks dropped — task definition broken (4)

These reach the verifier on every model and always score 0.0. None are agent
failures; the tasks are unwinnable as written.

### `ch_cli_status_checker` — instruction/verifier mismatch

- `instruction.md` tells the agent to write `clickhouse-client --query "SELECT version()"`.
- `tests/test_final_state.py` asserts the script contains the literal
  `clickhousectl cloud service get`.

Every model correctly follows the instructions and fails the assertion. 3 of 4
sub-tests pass; only `test_script_contains_correct_command` fails.

### `python_schema_migration_tool` — shared-cloud state pollution

The verifier runs `clickhouse-client` against the shared ClickHouse Cloud
instance configured via `CLICKHOUSE_HOST`. Bootstrap does no cleanup,
and the verifier never invokes the agent's `migrate.py`. As a result:

- A `users` table from prior runs (with columns `id, username, last_login`)
  shadows what the agent's `CREATE TABLE IF NOT EXISTS users (id, name)` would
  produce — `IF NOT EXISTS` silently no-ops against the polluted table.
- `schema_migrations` is empty because `migrate.py` is never executed.

Fails identically across all models.

### `python_schema_migration_tool_v2` — same shared-cloud pollution

`test_users_table_exists` passes (junk table left over), but
`test_email_column_exists` and `test_data_inserted` fail with
`Unknown expression identifier 'name'` because the leftover `users` schema is
`username, last_login`, not `name, email`. Verifier doesn't run `migrate.py`
either.

### `ch_python_async_insert` — environment bug

Verifier setup fails with `clickhouse-server is not listening on port 8123`
after a 60s `wait_for_port`. The Dockerfile CMD starts
`clickhouse-server --daemon` via `su -s /bin/bash clickhouse -c …`, but the
server doesn't reach a listening state in the available time. (The pattern that
works in `ch_python_github_archive` uses
`sudo -u clickhouse … || su …` plus a tighter foreground orchestration.)

## 5. Credential redaction in committed job logs

The agents ran `env | grep CLICKHOUSE` early in several trials, which captured
the live ClickHouse Cloud credentials into trajectory and stdout files.
Replacements applied across `clickhouse-benchmark/jobs/` before pushing:

| Original | Replacement | Files |
|---|---|---|
| ClickHouse Cloud password | `[REDACTED_CLICKHOUSE_PASSWORD]` | 12 |
| ClickHouse Cloud hostname | `[REDACTED_CLICKHOUSE_HOST]` | 32 |

Streaming-chunked partials of the hostname (the agent UI streamed it
character-by-character into the trajectory log) were also redacted via a wider
pattern.

Post-redaction sweep checked for `sk-*`, `sk-ant-*`, `AKIA*`, `ghp_*`,
`github_pat_*`, `xoxb-*`, `AIza*`, `hf_*`, `-----BEGIN ... PRIVATE KEY-----`,
JWT triplets, and bare credential-shaped values near `password|api_key|token|
secret|bearer`. All remaining matches are placeholders or variable references
(`your_password`, `your-instance.clickhouse.cloud`, `clickhousePassword`,
`os.environ.get(...)`).

> The leaked password sat unredacted on local disk before redaction and may
> exist in derivative artifacts. Rotate it.

## 6. Final state of `clickhouse-benchmark/`

- 18 task definitions (down from 24).
- `jobs/` containing 6 model runs from 2026-04-30 with redacted logs and
  recalculated `result.json`s.
- The 4 task-definition bugs above are not yet fixed; they were removed from
  the suite, not corrected. If they're needed back, the fixes are:
  - `ch_cli_status_checker`: change verifier to assert `clickhouse-client` and
    `SELECT version()` (matching `instruction.md`).
  - `python_schema_migration_tool[_v2]`: have the verifier `DROP TABLE IF EXISTS
    users[, schema_migrations]` first, then `subprocess.run(['python3',
    'migrate.py'])`, then assert.
  - `ch_python_async_insert`: replace the Dockerfile CMD with the
    `sudo -u clickhouse … || su …` pattern from `ch_python_github_archive`,
    and add a `wait_for_port 8123` loop in CMD before backgrounding.
